#### 구현 workflow
1. **Probe** — `scripts/probe_new_targets.py` 작성해 5개 target 모두 `lawSearch.do`(검색) + `lawService.do`(상세) 실제 호출, XML 응답을 `probe_out/`에 저장, 필드 이름·구조를 추출.
2. **필드 매핑 확정** — 각 target의 루트 태그, 검색 item 태그, detail root 구조, 상세 ID 파라미터 이름을 확정. ordin은 `ID=자치법규ID` (MST 아님). detc는 일부 케이스가 빈 응답(종국일자=0)이지만 정상 케이스는 populated.
3. **LawApiClient 확장** — `clients/law_api.py`에 5 target × (search + detail) = 10개 메서드 + 10개 dataclass 추가. `_get_xml` 헬퍼와 `MemoryCache`는 기존 재사용.
4. **Pydantic 스키마** — `models/schemas.py`에 search/detail Input/Output 모델 10개+ 추가. 기존 `InterpretationResultItem` 스키마는 실제 API 구조(`interpretation_id`, `case_name`, `inquiry_agency` 등)에 맞게 재작성.
5. **Tool handler 9개** — 신규 파일 8개 + `search_interpretation.py` 교체. 공통 패턴: search는 cache/filter/normalize, detail은 NotFound 예외 + 원문 그대로 반환.
6. **server.py** — 9개 `@mcp.tool` 등록. 도구 description은 한국어, 파라미터 가이드 포함.
7. **verify_tools.py 확장** — 신규 tool마다 search → detail 체인 회귀. `search → 첫 hit id → detail` 패턴을 hardcode ID 없이 자동 체이닝.
8. **AGENTS.md** — Tools 표를 6종 → 15종으로 업데이트, target 컬럼 추가, ordin `region` 필터의 generic-lookup 성격 명시.
